# Folder Guide

# How To Use This Documentation

Before starting a new feature:

1. Read TODO_NEXT.md
2. Check the roadmap
3. Read PROJECT_CONTEXT.md
4. Review SYSTEM_ARCHITECTURE.md
5. Follow DEVELOPMENT_RULES.md
6. Update documentation after implementation

Before making a major architectural decision:

1. Read SYSTEM_ARCHITECTURE.md
2. Review DECISIONS.md
3. Create a new decision record
4. Explain the reason clearly

Before changing UI:

1. Read DESIGN_SYSTEM.md
2. Follow the Korvix design philosophy
3. Preserve consistency
4. Avoid generic AI-generated UI patterns

Before changing APIs:

1. Read API_REFERENCE.md
2. Preserve backward compatibility
3. Document any new or changed endpoints

Before changing database schema:

1. Read DATABASE_SCHEMA.md
2. Prefer additive migrations
3. Avoid destructive changes
4. Document all changes

# Documentation Rule

Documentation is not optional.

Every significant feature, bug fix, architecture change, or production milestone should update the relevant documentation.

If a change affects how KorvixAI works, the documentation must reflect it.

---

# Maintenance Rule

At the end of every milestone, update:

* PROJECT_CONTEXT.md
* FEATURES.md
* CHANGELOG.md
* TODO_NEXT.md
* KORVIX_HISTORY.md

At the end of every major architecture change, update:

* SYSTEM_ARCHITECTURE.md
* DECISIONS.md
* DATABASE_SCHEMA.md
* API_REFERENCE.md
