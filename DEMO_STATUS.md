### Trading tab (`/trading/signals`)

- Nullable `setupGrade` / `confidence` — rows with no plan render `—`, never fake `D / 0%`

- Expanded row shows entry/target/stop, risk/reward, per-row provider, data-quality (`full | degraded | fallback`), asset type
- Provider routing on the backend: Binance → Yahoo → AlphaVantage → CoinGecko fallback chain

### Backend (Railway `worker-production-1345`)

## Known limitations

- **Sessions are local-only.** Server-side session sync exists in `backend/services/sessions/` but is feature-flagged off (`ENABLE_SESSIONS=false`). Closing the tab clears history.
- **No streaming.** `/chat` returns the full response as JSON. Acceptable for demo; SSE is the next phase.
- **Trading signals require flags.** Live data only when `ENABLE_TRADING_SIGNALS=true`, `ENABLE_TOOLS=true`, `ENABLE_MARKET_DATA=true` on Railway. Without flags the panel renders an honest "Trading signals service is disabled" state — never fakes prices.
- **Binance may be geo-blocked from Railway's region.** Crypto tickers fall back to CoinGecko (no per-symbol plan, only live price). The UI handles this gracefully — shows price + "no setup yet" instead of `D / 0%`.
- **Memory/Agent/Research subsystems are flag-off in prod.** Code is in `backend/services/memory|agent|research/`; turning them on requires env-var changes and is not part of the demo path.
- **No automated tests** for new pages or the trading mapper. Manual smoke is the verification path.
- **Sparkline data** is not generated by the backend; the UI gracefully skips rendering when absent.

## What NOT to touch before the demo

1. **`src/hooks/useChat.ts`** — `API_URL` is the canonical Railway host, must stay `worker-production-1345`. Do not refactor the friendly-error mapper or the timeout.
2. **`src/hooks/useTradingSignals.ts`** — same host + same mapper invariants. Do not narrow `setupGrade` / `confidence` back to non-nullable.
3. **`backend/routes/chat.py`** — `POST /chat` shape is what the frontend hard-codes against. No path/method change.
4. **`backend/api.py` CORS config** — regex + allow-list match the deployed Vercel preview/prod URLs.
5. **Railway env vars** — `ENABLE_TRADING_SIGNALS`, `ENABLE_TOOLS`, `ENABLE_MARKET_DATA` control the trading path. Flip on before demo only if the live data is part of the script.
6. **Railway start command** — unchanged since Phase 5.3 (per standing rule).
7. **`/chat/chat` rename** — never (per standing rule). The route is `/chat`.
8. **`STABLE_CHECKPOINT.md`** — source of truth for the canonical Railway URL. Don't bypass.

If anything in this list changes, the post-deploy health-check workflow will fail and the `railway/production-deploy` commit status will go red.

## Next recommended backend phase after the demo

**SSE streaming for `/chat`** — the single highest-impact upgrade.

Rationale:
- The hook + UI already render token-by-token (`assistantMessage.content` is a single string append today); converting to incremental append on each SSE chunk is a small frontend diff.
- Backend: add a sibling `POST /chat/stream` that wraps the existing OpenAI / Gemini call with `StreamingResponse` and yields `data:` frames. Keep `/chat` as-is for backward compat.
- Feature-flag: `ENABLE_CHAT_STREAMING=false` by default; flip on in Railway after smoke. Same rollback discipline as every prior phase.
- Zero blast-radius on trading, sessions, memory, agent, or research.

Follow-ons (in order):
1. **Server-side sessions on** (`ENABLE_SESSIONS=true`) — sessions already exist; just flag them on and add a `/workspaces/ensure_default` call to the frontend boot path.
2. **Memory write-through** (`ENABLE_NEW_MEMORY=true`) — chat turns persist to the memory layer; retrieved on next session.
3. **Agent loop on for trading queries** (`ENABLE_AGENT=true`) — function-calling drives `market_data_tool` for finance prompts.
4. **Research provider on** (`ENABLE_WEB_RESEARCH=true`) — Tavily-backed grounded citations for research prompts.

Each is one focused PR, one env var, one-line rollback.
